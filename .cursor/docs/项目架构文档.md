# SLG Server 项目架构文档

本文档从架构层面全面描述 SLG 游戏服务器的设计与实现，包括整体架构、模块划分、技术选型、核心系统实现等。

---

## 一、项目概述

### 1.1 项目定位

SLG Server 是一个基于 Java 21 + Spring Boot 3.3.0 的策略类游戏（SLG）服务端项目，采用多进程架构设计，支持分布式部署与横向扩展。

### 1.2 核心特性

- **多进程架构**：游戏服（Game）与场景服（Scene）分离，支持独立扩展
- **灵活通信**：支持 WebSocket 直连与 Redis Stream 路由两种 RPC 方式
- **高性能执行**：基于 Java 21 虚拟线程的统一执行器架构
- **数据驱动**：CSV 配置表 + MongoDB 文档存储 + Redis 缓存
- **事件驱动**：全局事件总线 + 跨服事件传递 + 进度管理系统
- **完整框架**：实体缓存、持久化、AOI 视野系统、同步总线等

### 2.3 核心数据流

```
客户端请求流程：
Client → WebSocket → MessageCodec → MessageHandler → Facade → Service → Manager

内部 RPC 调用流程：
Caller → RpcProxy → (WebSocket直连/Redis路由) → RpcFacade → RpcMethod → Service

事件驱动流程：
EventPublisher → EventBusManager → EventListenerWrapper → @EventListener 方法

数据持久化流程：
业务层 → EntityCache → AsyncPersistenceService → MongoDB/MySQL
```

## 三、核心系统详解

### 3.1 网络通信系统 (slg-net)

#### 3.1.1 协议设计

**线缆格式**：`Length(VarInt) + MsgId(VarInt) + Body`

**编解码特性**：
- 使用 VarInt 编码协议号和长度，减少包体积
- 基于类型协议号的二进制编解码（非 JSON）
- 支持 Nest、List、Map、Enum 等复杂类型

**协议注册** (`message.yml`)：
```yaml
messages:
  rpc:
    - 101, IM_RpcRequest
    - 102, IM_RpcRespone
  login:
    - 1001, CM_LoginReq
    - 1002, SM_LoginResp
  scene:
    - 1100, SM_EnterScene
    - 1101, CM_LoadSceneFinish
```

**协议号分配规则**：
- `0`：null（系统保留）
- `1-50`：基础类型（Byte、Int、String 等）
- `100-999`：内部消息（RPC、Socket 等）
- `1000+`：客户端消息（按模块预留 100 个区间）

#### 3.1.2 WebSocket 服务器

**实现位置**：`slg-net/socket/server/WebSocketServer.java`

**核心特性**：
- 基于 Netty 的高性能 WebSocket 实现
- 支持自定义 ChannelInitializer
- Boss/Worker 线程模型可配置
- 优雅关闭机制

**Pipeline 配置**：
```
WebSocketFrameToByteBufDecoder
ByteBufToWebSocketFrameEncoder
MessageDecoder
MessageEncoder
WebSocketMessageHandler
```

#### 3.1.3 RPC 框架

**核心组件**：

| 组件 | 职责 |
|------|------|
| `RpcProxyManager` | 管理 RPC 接口代理，处理远程调用 |
| `RpcFacade` | 处理 RPC 请求和响应消息 |
| `RpcCallBackManager` | 管理 RPC 回调和超时 |
| `RpcMethodInjector` | 扫描并注入 RPC 方法 |
| `RpcRemoteManager` | 管理远程服务连接 |

**RPC 调用流程**：
```
Caller → @RpcRef 注入的代理
        ↓
   判断是否本地调用 (SingleStart)
        ↓
   [本地] 直接调用方法
   [远程] → IM_RpcRequest → WebSocket/Redis → 目标服务器
        ↓
        ↓
   目标服务器 → RpcFacade → MethodHandle.invoke(目标方法)
        ↓
   返回结果 → IM_RpcRespone → 原路返回 → Future.complete()
```

**路由策略**：

| 路由方式 | 实现类 | 适用场景 |
|---------|---------|---------|
| `ServerIdRoute` | 按 serverId 直连 | 单 Game 服或少量 Game 服 |
| `PlayerCurrentSceneRoute` | 按玩家当前场景路由 | 场景相关 RPC 调用 |
| `PlayerGameRoute` | 按玩家所在 Game 服路由 | 玩家数据相关 RPC |
| `RedisRoute` | Redis Stream 转发 | 多 Game 服跨服调用 |

#### 3.1.4 消息处理器

**Facade 注解处理**：
```java
@Component
public class LoginFacade {

    @MessageHandler
    public void login(NetSession session, CM_LoginReq req) {
        // 两参数：未绑定 Player 阶段使用
    }

    @MessageHandler
    public void gainTask(NetSession session, CM_GainTask req, Player player) {
        // 三参数：已绑定 Player，框架自动注入
    }
}
```

**线程模型**：
- 两参数方法：在 Login/System/Scene 等单链中执行
- 三参数方法：在 Player 虚拟线程链中执行

### 3.2 执行器架构 (slg-common/executor)

#### 3.2.1 设计理念

基于 Java 21 虚拟线程的统一任务调度系统，替代传统线程池模型。

**核心优势**：
- 轻量级：虚拟线程创建成本低，可创建数百万个
- 阻塞友好：阻塞操作不会占用平台线程
- 类型安全：编译期区分多链/单链，防止调用错误

#### 3.2.2 核心组件

| 组件 | 职责 |
|------|------|
| `KeyedVirtualExecutor` | 按 TaskKey(module, id) 分链串行执行 |
| `GlobalScheduler` | 全局定时调度（2个平台线程 + 虚拟线程池） |
| `VirtualExecutorHolder` | 共享虚拟线程池（`newVirtualThreadPerTaskExecutor`） |
| `MultiExecutor` | 多链模块封装，必须传 key |
| `SingleExecutor` | 单链模块封装，不传 key |
| `Executor` | 统一入口，通过反射自动初始化各模块执行器 |

#### 3.2.3 模块分类

```java
// 多链模块（按 key 隔离）
Executor.Player.execute(playerId, () -> { ... });           // 按玩家 ID 分链
Executor.Persistence.execute(entityId, () -> { ... });     // 按实体 ID 分链
Executor.SceneNode.execute(nodeId, () -> { ... });        // 按节点 ID 分链

// 单链模块（全局串行）
Executor.System.execute(() -> { ... });                   // 所有系统任务串行
Executor.Login.execute(() -> { ... });                   // 所有登录任务串行
Executor.Scene.execute(() -> { ... });                   // 所有场景任务串行
```

#### 3.2.4 虚拟线程使用规范

**允许的阻塞**：
- `CompletableFuture.join()` 在多链模块中
- `ReentrantLock.lock()` / `unlock()`
- I/O 操作（网络、磁盘等）

**禁止的操作**：
- 使用 `synchronized`（会 pin 载体线程）
- 在单链模块中使用 `.join()`（会阻塞整条链）
- 自行创建线程池

**RPC 同步调用规范**：

```java
// ✅ 正确：多链模块使用 join() 安全
try {
    int result = sceneRpcService.enterScene(serverId, playerId).join();
} catch (CompletionException e) {
    // 处理异常
}

// ❌ 错误：单链模块禁止使用 join()
Executor.Scene.execute(() -> {
    int result = someRpc.doSomething().join(); // 会阻塞所有场景任务
});

// ✅ 正确：单链模块使用异步回调
Executor.Scene.execute(() -> {
    someRpc.doSomething().whenComplete((result, error) -> {
        Executor.Scene.execute(() -> handleResult(result, error));
    });
});
```

### 3.3 事件系统 (slg-common/event)

#### 3.3.1 EventBusManager

**核心职责**：
- 管理事件类型到监听器的映射
- 同步分发事件到所有注册的监听器
- 线程安全的 `ConcurrentHashMap` 存储

**使用方式**：
```java
// 发布事件
EventBusManager.getInstance().publishEvent(new PlayerLoginEvent(playerId));

// 监听事件
@EventListener
public void onPlayerLogin(PlayerLoginEvent event) {
    // 处理登录事件
}
```

#### 3.3.2 自动注入

`EventListenerInjector` 作为 BeanPostProcessor 自动扫描：
- 带有 `@EventListener` 注解的方法
- 事件类型（第一个参数类型）
- 优先级（@EventListener 中的 order）

启动时自动注册到 EventBusManager。

### 3.4 进度管理系统 (slg-shared-modules/progress)

#### 3.4.1 设计理念

基于事件驱动的进度管理系统，支持任务、成就等场景。

**核心特性**：
- 自动扫描 IProgressEvent 和 IProgressCondition
- 支持多种进度拥有者（玩家、公会等）
- 进度更新与完成回调互斥执行
- 纯缓存设计，不涉及持久化

#### 3.4.2 核心组件

| 组件 | 职责 |
|------|------|
| `ProgressManager` | 进度管理器，注册进度、触发事件 |
| `IProgressCondition<T, E>` | 进度条件接口，定义如何处理事件 |
| `IProgressEvent` | 进度事件接口，携带拥有者信息 |
| `IProgressTable` | 进度配置表接口，读取 CSV 配置 |
| `ProgressMeta` | 进度元数据，存储当前进度与状态 |
| `IProgressTypeTransform` | 进度类型转换器 |

#### 3.4.3 使用示例

```java
// 定义进度条件
@Component
public class LoginCountCondition implements IProgressCondition<Player, PlayerLoginEvent> {

    @Override
    public void onEvent(PlayerLoginEvent event, ProgressMeta meta) {
        int count = meta.getProgress() + 1;
        meta.setProgress(count);
    }

    @Override
    public long getFinishProgress() {
        return 5; // 登录 5 次完成
    }

    @Override
    public void init(Player owner, ProgressMeta meta) {
        meta.setProgress(0);
    }
}

// 注册进度
progressManager.registerProgress(
    ProgressOwnerEnum.PLAYER,
    playerId,
    player,
    mainTaskTable,
    progressMeta
);

// 触发事件（会自动更新进度）
EventBusManager.getInstance().publishEvent(new PlayerLoginEvent(playerId));
```

### 3.5 实体缓存系统 (slg-support/entity/cache)

#### 3.5.1 设计理念

三层缓存架构：本地缓存（Caffeine）+ 异步持久化（Write-Behind）+ 数据库。

**核心特性**：
- 缓存优先，减少数据库访问
- 异步持久化，不阻塞业务线程
- 支持字段级更新
- 自动生命周期管理

#### 3.5.2 核心组件

| 组件 | 职责 |
|------|------|
| `EntityCache<T>` | 单一类型的缓存实例 |
| `EntityCacheManager` | 缓存管理器，管理所有类型缓存 |
| `AsyncPersistenceService` | 异步持久化服务 |
| `WriteBehindBuffer` | 写后缓冲区 |
| `CacheFlushLifeCycle` | 关停时刷盘生命周期 |

#### 3.5.3 使用示例

```java
@Component
public class PlayerService {

    @EntityCacheInject
    private EntityCache<PlayerEntity> playerCache;

    // 查询
    PlayerEntity player = playerCache.findById(playerId);

    // 插入
    PlayerEntity newPlayer = new PlayerEntity();
    playerCache.insert(newPlayer);

    // 更新
    player.setLevel(10);
    playerCache.save(player);

    // 字段更新
    player.setGold(100);
    playerCache.saveField(player, "gold");

    // 删除
    playerCache.deleteById(playerId);
}
```

#### 3.5.4 缓存配置

```java

    writeBatchSize = 100    // 批量写入大小
)

    // ...
}
```

### 3.7 AOI 视野系统 (slg-scene/scene/aoi)

#### 3.7.1 设计理念

基于九宫格算法的 Area of Interest（兴趣区域）系统，实现高效的视野管理。

**核心特性**：
- 多层级网格支持
- 延迟删除机制（4秒）
- 批量广播优化
- 军队独立视野裁剪

#### 3.7.2 核心组件

| 组件 | 职责 |
|------|------|
| `AoiController` | AOI 控制器，管理视野更新 |
| `MultiGridContainer` | 多层网格容器 |
| `AoiGrid` | 单层网格 |
| `GridLayer` | 网格层级（缩放级别） |
| `Watcher` | 观察者（玩家） |

#### 3.7.3 AOI Tick 流程

```
Step 1: 并行计算所有玩家的视野变化
  ├─ 收集新视野内的所有 node
  ├─ 对比旧视野，标记需要遗忘的 node
  └─ 对比旧视野，标记新增的 node

Step 2: 按 node 聚合，批量广播新增消息
  └─ 一次广播发送给所有相关玩家

Step 3: 每4秒执行，批量广播遗忘消息
  └─ 延迟删除，避免频繁抖动
```

#### 3.7.4 使用示例

```java
// 提交视野更新任务（当玩家镜头移动时调用）
aoiController.submitWatchTask(playerId, position, gridLayer);

// AOI Tick 在场景主循环中定时执行
public void sceneTick() {
    aoiController.aoiTick();  // 处理视野更新
    aoiController.armyTick();  // 更新军队位置
}
```

### 3.9 同步总线 (slg-net/syncbus)

#### 3.9.1 设计理念

轻量级跨进程实体字段同步系统，实现 Holder 端到 Cache 端的字段值传送。

**核心特性**：
- 复用 RPC 框架
- 本地调用零网络开销
- 支持字段限流
- 支持编解码器

#### 3.9.2 核心组件

| 组件 | 职责 |
|------|------|
| `SyncBus` | 同步总线主类 |
| `ISyncHolder` | 数据源端接口 |
| `ISyncCache` | 数据接收端接口 |
| `@SyncEntity` | 实体同步注解 |
| `@SyncField` | 字段同步注解 |

#### 3.9.3 使用示例

```java
// Holder 端（Game）
@SyncEntity(SyncModule.PLAYER)
public class PlayerEntity implements ISyncHolder {

    @SyncField(syncInterval = 5)  // 5秒内最多发送一次
    private long allianceId;

    @SyncField(syncInterval = 0)  // 不限流
    private int level;

    @Override
    public long getSyncId() {
        return playerId;
    }

    @Override
    public List<Integer> syncTargetServerIds() {
        // 返回需要同步的目标服务器 ID 列表
    }
}

// Cache 端（Scene）
@SyncEntity(SyncModule.PLAYER)
public class ScenePlayerEntity implements ISyncCache {

    @SyncField
    private long allianceId;

    @SyncField
    private int level;

    @Override
    public long getSyncId() {
        return playerId;
    }
}

// 业务调用
playerEntity.setAllianceId(newAllianceId);
SyncBus.sync(playerEntity, PlayerEntity.Fields.allianceId);
```

### 3.10 跨服事件 (slg-net/crossevent)

#### 3.10.1 设计理念

零侵入的跨服事件传递系统，自动将本地事件转发到目标服务器。

**核心特性**：
- 对 EventBusManager 零侵入
- 支持多种路由策略
- 防循环转发机制
- 自动监听器注册

#### 3.10.2 路由注解

| 注解 | 路由策略 |
|------|---------|
| `@RouteServer` | 按 serverId 路由 |
| `@RoutePlayerGame` | 按玩家所在 Game 服路由 |
| `@RoutePlayerMainScene` | 按玩家主场景路由 |
| `@RoutePlayerCurrentScene` | 按玩家当前场景路由 |

#### 3.10.3 使用示例

```java
// 定义事件 VO（位于 slg-net）
public class HeroLevelUpEventVO implements IEvent {
    @RoutePlayerGame  // 路由注解
    private long playerId;
    private int heroId;
    private int level;
}

// 业务事件实现
public class HeroLevelUpEvent implements IProgressEvent, ICrossServerEvent {

    @Override
    public HeroLevelUpEventVO toCrossEvent() {
        HeroLevelUpEventVO vo = new HeroLevelUpEventVO();
        vo.setPlayerId(player.getId());
        vo.setHeroId(heroId);
        vo.setLevel(level);
        return vo;
    }
}

// 发布事件（自动跨服传播）
EventBusManager.getInstance().publishEvent(new HeroLevelUpEvent(...));
```

## 四、数据存储架构

### 4.1 MongoDB（主业务存储）

**使用场景**：玩家、英雄、任务、场景实体等游戏主数据

**数据模型**：
```javascript
{

  "updateTime": ISODate("2025-01-01T12:00:00Z")
}
```

**索引策略**：
- 主键索引：`_id`
- 查询索引：根据查询需求创建复合索引
- TTL 索引：定期清理过期数据

### 4.2 Redis（缓存 + RPC 路由）

**业务缓存**：
- Key 格式：`cache:{modulePrefix}:{entityId}`
- 数据结构：Hash
- 序列化：JSON（基于 Jackson）
- 过期策略：LRU + TTL

**RPC 路由（独立 Redis 实例）**：
- Stream Key：`rpc:route:{serverId}`
- 响应 Key：`rpc:route:resp:{callerServerId}`
- MaxLen：10000（防止目标离线时堆积）

### 4.3 MySQL（日志服务）

**使用场景**：slg-log 模块的日志存储

**数据模型**：
```sql
CREATE TABLE game_log (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  player_id BIGINT,
  log_type VARCHAR(50),
  log_content TEXT,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_player_id (player_id),
  INDEX idx_create_time (create_time)
);
```

### 4.4 Elasticsearch（日志检索）

**使用场景**：集中日志存储与全文检索

**索引模板**：
```json
{
  "index_patterns": ["game-log-*"],
  "template": {

```

## 六、性能优化策略

### 6.1 网络层优化

1. **二进制协议**：减少包体积，提升解析速度
2. **VarInt 编码**：压缩整数传输
3. **批量广播**：按节点聚合同一消息发送给多个玩家
4. **连接复用**：RPC 连接池，避免频繁建立连接

### 6.2 执行层优化

1. **虚拟线程**：降低上下文切换成本，支持高并发
2. **任务分链**：避免全局锁竞争
3. **异步持久化**：不阻塞业务线程
4. **定时任务合并**：统一调度器，减少线程数

### 6.3 数据层优化

1. **三级缓存**：本地缓存 + Redis + 数据库
2. **批量操作**：批量读写，减少网络往返
3. **字段级更新**：只更新变化的字段
4. **索引优化**：根据查询模式建立复合索引

### 6.4 AOI 系统优化

1. **九宫格算法**：O(1) 的空间查询
2. **延迟删除**：避免视野边界频繁抖动
3. **层级网格**：根据缩放级别选择不同粒度
4. **批量广播**：减少网络消息数量
