# ADR-002: 虚拟线程替代传统线程池

## 状态

已采纳

## 背景

项目原有多个模块各自创建 `ScheduledExecutorService` / `ThreadPoolExecutor`（如 PlayerExecutorService、SystemExecutorService、RpcExecutor 等），线程资源分散、调度逻辑不统一。Java 21 提供虚拟线程（Project Loom），可大幅降低上下文切换与内存占用。

## 决策

统一迁移到**虚拟线程**架构：
- 底层：`KeyedVirtualExecutor`（按 TaskKey 分链串行）+ `GlobalScheduler`（2 个平台调度线程）+ `VirtualExecutorHolder`（共享虚拟线程池）
- 上层：`MultiExecutor`（多链，按 key 分链）与 `SingleExecutor`（单链）封装
- 业务层：通过 `Executor.Player` / `Executor.System` 等静态字段调用，禁止自建线程池

## 理由

1. Java 21 虚拟线程可降低「一请求一线程」的上下文切换与内存开销
2. 统一调度入口，减少线程池碎片化与并发 bug
3. `TaskModule` 枚举 + 反射自动初始化，新增模块只需添加枚举值和对应静态字段
4. 类型系统在编译期防止误用（多链必须传 key，单链不接受 key）

## 后果

- **禁止 `synchronized`**：虚拟线程中 synchronized 会 pin 载体平台线程，必须改用 `ReentrantLock`
- **单链禁止 `.join()`**：单链模块若同步等待 RPC 会阻塞整条链（见 ADR-003）
- 已删除的旧类：各模块的 Executor、PlayerExecutorService、RpcExecutor、PersistenceThreadPool 等
