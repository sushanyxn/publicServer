## 1.1 模块结构

项目采用 Maven 多模块结构，共 12 个子模块，职责划分如下：

| 模块 | 职责 | 主要依赖 |
|------|------|----------|
| **slg-common** | 公共基础：工具类、通用常量 | SLF4J、Jackson、Spring Context、Lombok |
| **slg-net** | 网络层：WebSocket 服务端/客户端、协议编解码、RPC 框架、SyncBus 跨进程同步、跨服事件、ZK 集成 | Netty、Curator、ClassGraph、CGLIB、SnakeYAML |
| **slg-redis** | Redis 封装：连接与模板配置 | spring-boot-starter-data-redis、commons-pool2 |
| **slg-support** | 支撑层：实体缓存、生命周期管理、CSV 配置表加载、MongoDB/JPA 集成 | Spring Data MongoDB、Spring Data JPA、Caffeine、fastutil、Commons CSV |
| **slg-shared-modules** | 共享模块：战斗结算、数值计算、属性/进度等完整系统能力（含读表与协议转化），供 game/scene 共用 | slg-common、slg-net、slg-support |
| **slg-scene** | 场景服：场景逻辑与节点管理 | slg-support、slg-shared-modules、slg-net、MongoDB、Redis、Curator |
| **slg-game** | 游戏服：核心业务逻辑 | slg-support、slg-shared-modules、slg-net、MongoDB、Redis、Curator |
| **slg-singlestart** | 单体启动：将 Game 与 Scene 合并到同一进程运行，便于开发与调试 | slg-game、slg-scene |
| **slg-robot** | 机器人/压测客户端：模拟客户端连接与协议交互 | slg-net、slg-support |
| **slg-log** | 日志服务：日志上报、检索与统计 | Spring Web、Spring Security、JJWT、JPA、MySQL、Elasticsearch |
| **slg-framework-test** | 框架测试：底层框架的集成测试 | Testcontainers、JPA、MySQL |

## 二、分项选型与横向对比

### 2.1 语言与运行时：Java 21

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Java 21 (当前)** | 高 | LTS；虚拟线程、Record、模式匹配等，与 Spring Boot 3.x 对齐。 |
| Java 17 | 高 | LTS，生态成熟；无虚拟线程等新特性。 |
| Java 11 | 中 | 仍广泛使用；Spring Boot 3.x 要求 Java 17+，不可用。 |

#### 游戏服场景下的考量

- **Java 21**：虚拟线程可降低「一请求一线程」的上下文切换与内存占用，适合高并发长连接；ZGC/G1 对低延迟与吞吐的调优成熟；与 Netty、Spring Boot 3.x 全栈兼容。
- **Java 17**：若无虚拟线程需求，LTS 支持与生态同样充足，可作为保守选择。
- **其他语言**：Kotlin 与 Java 互操作好但增加语法层；Go/Rust 需重写或双栈维护，游戏服主栈仍以 JVM 生态为主。

#### 横向对比小结

| 维度 | Java 21 (当前) | Java 17 | Java 11 |
|------|----------------|---------|---------|
| **LTS 与支持** | 是，长期支持 | 是 | 是，但较老 |
| **Spring Boot 3.x** | 支持 | 支持 | 不支持 |
| **虚拟线程** | 有 | 无 | 无 |
| **GC 与调优** | ZGC/G1 成熟 | 同左 | G1 为主 |
| **游戏服适用** | 高并发、长连接、低延迟友好 | 同样适用 | 版本过旧不推荐 |

#### 选型评价

当前选用 **Java 21** 合理：与 Spring Boot 3.5 一致、具备虚拟线程与现代 GC，满足游戏服高并发与可维护性需求；若部署环境仅支持 Java 17，可退守 17 而不影响主架构。

### 2.2 构建工具：Maven

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Maven (当前)** | 高 | 多模块与 BOM 管理清晰，团队熟悉度高。 |
| Gradle | 高 | 构建更快、DSL 灵活；迁移成本与习惯变更。 |

#### 对比说明

- **Maven**：XML 声明式、生命周期统一；dependencyManagement 与 BOM 在本项目中已很好支撑多模块与版本收敛；CI/IDE 支持普遍；构建速度在模块数适中时可接受。
- **Gradle**：增量构建与缓存更激进，大项目构建更快；DSL 灵活、脚本化强；需学习 Groovy/Kotlin DSL，且与现有 pom 体系不同。

#### 横向对比小结

| 维度 | Maven (当前) | Gradle |
|------|--------------|--------|
| **配置方式** | XML，声明式 | Groovy/Kotlin DSL，可脚本化 |
| **多模块/BOM** | 原生支持，当前已用 | 支持，需重写脚本 |
| **构建速度** | 中等 | 通常更快（增量、缓存） |
| **学习与迁移** | 团队已熟悉 | 需迁移成本与规范统一 |
| **CI/生态** | 广泛 | 广泛 |

#### 选型评价

**Maven** 满足当前多模块与 BOM 管理需求，无强制替换必要；若后续模块数或构建耗时明显增加，可再评估 Gradle 迁移，届时需统一 DSL 风格与 CI 脚本。

---

### 2.3 应用框架：Spring Boot 3.5.12

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Spring Boot 3.x (当前)** | 高 | 与 Java 21、Jakarta 命名空间一致；生态完整。 |
| Spring Boot 2.x | 低 | 基于 Java 8/11 与 javax，与当前 JDK 与依赖不统一。 |
| 纯 Netty/自研容器 | 中 | 更轻、可控；需自行解决配置、生态与运维习惯。 |

#### 对比说明

- **Spring Boot 3.x**：提供配置、依赖注入、生命周期、与 MongoDB/Redis/JPA 等集成；游戏服/场景服仅用其「容器与配置」层，长连接与协议仍由 Netty 承担，职责清晰。
- **Quarkus / Micronaut**：启动快、内存占用小，更适合 Serverless 或容器冷启动；游戏服多为常驻进程，Spring 生态与团队习惯优势更大。
- **纯 Netty 自研**：无 Spring 依赖，包体更小；但配置、健康检查、与 Redis/Mongo 等组件的集成需自建，运维与可观测性成本高。

#### 横向对比小结

| 维度 | Spring Boot 3.x (当前) | Quarkus/Micronaut | 纯 Netty 自研 |
|------|------------------------|-------------------|----------------|
| **与 Java 21** | 原生支持 | 支持 | 无关 |
| **与 Netty** | 可并存，各司其职 | 可并存 | 以 Netty 为主 |
| **数据/缓存集成** | 成熟（Mongo/Redis/JPA） | 有支持 | 需自建 |
| **启动与内存** | 中等 | 更优 | 最轻 |
| **运维与可观测** | Actuator、生态成熟 | 有对应能力 | 需自建 |

#### 选型评价

**Spring Boot 3.5** 作为基础框架合适：与 Netty 分工明确（Spring 管配置与集成，Netty 管长连接与协议），生态与可维护性好；在无强需求（如极致冷启动、极小镜像）前提下，不引入 Quarkus/自研容器，避免重复造轮子。

### 2.4 网络层：Netty WebSocket + 自定义 RPC

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Netty + 自研 RPC (当前)** | 高 | 与业务协议（message.yml、编解码）深度绑定，可控性高；支持直连与 Redis 路由。 |
| gRPC | 高 | 标准、多语言；与现有 packet/IM_/CM_ 体系重构量大。 |
| Dubbo/RSocket | 中 | 功能强；引入重量级依赖与运维，与当前轻量 RPC 定位不同。 |

#### 对比说明

- **Netty + 自研 RPC**：客户端与游戏服走 WebSocket + 自定义二进制协议（message.yml 注册、MessageCodec 编解码）；Game↔Scene 内部 RPC 走同一套消息体系，支持直连（`ServerIdRoute`，走 WebSocket）与 Redis Stream 路由（`RedisRoute`，跨服转发），路由方式由 `@RpcMethod(routeClz = ...)` 声明，与业务强绑定、无额外协议层。
- **gRPC**：HTTP/2 + Protobuf，多语言、流式、标准化；但需将现有 CM_/SM_/IM_/VO/Event 与 message.yml 映射为 .proto 与生成代码，客户端若为 Unity/UE 需单独适配，改造面大。
- **Dubbo**：服务注册发现、负载均衡、治理完善；偏向微服务 RPC，与当前「按 serverId 直连/Redis 路由」的轻量模型不同，且引入 ZK/Nacos 等注册中心与 Dubbo 协议栈。
- **RSocket**：响应式、多路复用；生态与游戏行业采用面不如 Netty 广，协议与现有 message 体系不统一。

#### 横向对比小结

| 维度 | Netty + 自研 RPC (当前) | gRPC | Dubbo |
|------|-------------------------|------|-------|
| **与现有协议** | 直接复用 message.yml/Codec | 需 .proto 与重构 | 需适配 Dubbo 协议 |
| **客户端** | 同一套协议，前端/Unity 可复用 | 需各端生成 stub | 通常服务端间 |
| **直连 + Redis 路由** | 已实现 | 需自建路由层 | 有注册中心，模型不同 |
| **多语言** | 需各端自实现 | 原生多语言 | 以 JVM 为主 |
| **运维与依赖** | 轻量 | 轻量 | 较重 |

#### 选型评价

当前 **Netty + 自定义 RPC + message.yml** 与项目协议与路由需求匹配：客户端与内部 RPC 共用一套消息注册与编解码，直连与 Redis 路由已落地，保持现状合理。若未来有多语言服务或强标准化需求，可单独对「内部 RPC」试点 gRPC，与现有 WebSocket 客户端协议并存。

### 2.5 协议序列化与 JSON（自定义二进制 + Jackson）

本项目**协议不使用 JSON**，而是使用 **自定义二进制编解码**：线缆格式为 `Length(VarInt) + MsgId(VarInt) + Body`，Body 由 `MessageCodec` 按类型协议号（message.yml 注册）写入 VarInt、基本类型、字符串、集合及注册的消息类，实现紧凑二进制序列化。message.yml 中注册的类型包括：**CM_**（客户端→服务端）、**SM_**（服务端→客户端）、**IM_**（内部消息/RPC）、**VO**（值对象/嵌套结构）、**Event**（事件类）。**Jackson** 用于协议以外的 JSON 场景（配置、SyncBus 跨进程字段同步的 fieldData、通用 JsonUtil 等）。

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **自定义二进制 MessageCodec (当前)** | 高 | 与 message.yml、协议号一一对应；VarInt + 类型标记，包体紧凑、无 JSON 开销。 |
| **Jackson（仅非协议）** | 高 | 配置、内部 JSON 等；不参与协议线缆。 |
| Protobuf | 高 | 体积与性能更优；需 .proto 与各端生成，与现有 Codec 双轨。 |
| Kryo/Fastjson2 | 中 | 性能好；协议若改用需替换整套 Codec，生态/安全需考量。 |

#### 对比说明

- **MessageCodec（协议）**：二进制、按字段类型与注册类型号编解码，与 message.yml 和 Netty ByteBuf 管道一致；WebSocket 与 Redis Stream 路由均使用同一套 `MessageWireCodec`（Length + MsgId + Body），非 JSON。
- **Jackson**：仅用于非协议场景（如 SyncBus 的 fieldData、ZK 配置 JSON、通用 JsonUtil）；不与客户端/内部协议线缆混用。
- **Protobuf**：若希望协议更小、更快或跨语言统一描述，可对部分协议试点；需与现有 MessageCodec 并存（按协议号分支），而非替代 Jackson（协议本来就不用 Jackson）。
- **Kryo/Fastjson2**：可作为内部 RPC 或缓存的序列化；若用于协议需重写编解码并与 message.yml 解耦，改动大。

#### 横向对比小结

| 维度 | 自定义二进制 MessageCodec (当前) | Protobuf | 若协议用 JSON/Jackson |
|------|--------------------------------|----------|------------------------|
| **协议线缆格式** | VarInt + 类型号 + 二进制字段 | 二进制，.proto 定义 | 文本，包体大、解析慢 |
| **与 message.yml** | 直接对应 | 需 .proto 映射 | 可对应，但不利于带宽 |
| **包体与性能** | 紧凑、无文本解析 | 更优 | 差于二进制 |
| **跨语言** | 各端需实现相同 Codec | 原生多语言 | 各端 JSON 库即可 |

#### 选型评价

**协议采用自定义二进制（MessageCodec）** 正确：与 message.yml 和现有管道一致，包体小、无 JSON 开销。**Jackson** 仅负责协议之外的 JSON，职责清晰。若后续对带宽或延迟有更高要求，可对热点协议试点 **Protobuf**，与现有 MessageCodec 按协议号并存，而不是「用 Jackson 做协议」。

### 2.6 主业务存储：MongoDB

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **MongoDB (当前)** | 高 | 文档模型适合游戏实体、灵活 schema；Spring Data 集成成熟。 |
| MySQL/PostgreSQL | 高 | 关系型、事务强；对复杂嵌套与频繁 schema 变更支持不如文档库自然。 |
| Redis 持久化 | 低 | 偏缓存与路由，不做主持久化。 |

**结论**：MongoDB 作为 game/scene 主存合理；与 support 层实体缓存、生命周期配合良好。

### 2.6.1 MongoDB 与 MySQL 游戏服场景对比与选型评价

游戏服数据通常具备：以玩家/实体为中心、单文档内嵌套多（背包、英雄、任务等）、版本迭代导致字段频繁增减、读写以「按玩家/实体整存整取」为主、跨表关联查询相对较少等特点。下面对两种数据库在游戏服场景下的优劣势做集中对比，并给出本项目选型评价。

#### 游戏服场景下 MongoDB 的优势

| 维度 | 说明 |
|------|------|
| **数据模型** | 文档（BSON）与「一个玩家/实体一张大对象」的思维一致，天然支持嵌套数组与子对象（如英雄列表、背包格子、任务进度），无需拆多表、少 JOIN。 |
| **Schema 灵活** | 版本迭代中新增/废弃字段、增减数组元素无需 DDL 变更，适合策划需求频繁调整的配置与存档。 |
| **读写模式** | 按 _id（如玩家 ID）读整文档、改后写回，与游戏服「内存实体落盘」的节奏匹配；单文档原子更新即可覆盖大部分存档场景。 |
| **扩展与运维** | 分片（Sharding）按玩家 ID 等键水平切分简单；副本集提供高可用与读扩展。 |
| **与内存模型衔接** | 实体从 MongoDB 反序列化到内存对象、再写回，结构一一对应，与 slg-support 的实体缓存、生命周期配合顺畅。 |

#### 游戏服场景下 MySQL 的优势

| 维度 | 说明 |
|------|------|
| **事务与一致性** | 多行/多表 ACID 事务、外键约束，适合强一致、强关联的结算（如订单、充值、跨表扣费）。 |
| **复杂查询与统计** | 多表 JOIN、聚合、报表、运营查询（如全服排行、跨玩家统计）用 SQL 表达直接，索引与执行计划成熟。 |
| **运维与生态** | 运维经验普及、备份与恢复工具成熟；与现有管理后台、BI、日志库技术栈统一。 |
| **存储与成本** | 在高度结构化、低嵌套场景下，行存更省空间；对「扁平表 + 关联」型数据更友好。 |

#### 在本项目中的角色分工

- **MongoDB**：承担 **game / scene 主业务存储**（玩家、英雄、任务、场景相关实体等）。数据形态以「单玩家/实体一大文档」为主，读写以整文档或文档内局部更新为主，与文档模型和 slg-support 实体缓存设计一致。
- **MySQL**：承担 **日志侧**（slg-log）的持久化，包括日志表等。强事务、关联查询、报表需求集中在此，用 JPA + 关系型更合适。

#### 最终选型评价

| 评价项 | 说明 |
|--------|------|
| **主存选型** | 游戏服主存选用 **MongoDB** 合理：贴合游戏实体与存档的数据形态，schema 灵活利于版本迭代，与当前「实体缓存 + 按玩家落盘」的架构匹配；无需为游戏主流程引入多表 JOIN 与复杂事务。 |
| **日志选型** | 日志选用 **MySQL + JPA** 合理：满足事务、关联与报表需求，团队与运维熟悉度高，与主业务存储职责分离清晰。 |
| **混合使用** | 本项目采用 **MongoDB（主业务）+ MySQL（管理与日志）** 的混合方案，各司其职，无需二选一；若未来出现「跨库统计」（如从 MongoDB 同步汇总到 MySQL/数仓），可通过离线同步或 CDC 等方式扩展，而不必推翻现有主存选型。 |

### 2.7 关系型与管理库：MySQL + JPA

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **MySQL + JPA (当前)** | 高 | 用于 slg-log 等日志场景，事务与查询习惯统一。 |
| MyBatis/MyBatis-Plus | 高 | SQL 可控；项目已用 JPA，无强制替换理由。 |
| PostgreSQL | 高 | 功能更强；若当前无跨库需求可维持 MySQL。 |

#### 对比说明

- **MySQL + JPA**：日志侧以「实体 + 事务 + 关联查询」为主，JPA 与 Spring Data 集成好，DDL 与迁移可配合 Flyway/Liquibase；团队对 MySQL 运维熟悉度高。
- **MyBatis/MyBatis-Plus**：SQL 显式可控、复杂查询与优化直接；当前无大量复杂 SQL 与动态 SQL 需求，JPA 已覆盖；若后续报表与统计 SQL 增多，可在局部模块引入 MyBatis 而不必全盘替换。
- **PostgreSQL**：JSON/JSONB、窗口函数、扩展丰富；若未来管理端有强分析或 JSON 字段查询需求，可评估迁移或读写分离到 PostgreSQL。

#### 横向对比小结

| 维度 | MySQL + JPA (当前) | MyBatis-Plus | PostgreSQL |
|------|--------------------|--------------|------------|
| **使用场景** | slg-log | 可作 JPA 补充 | 可作 DB 替代/补充 |
| **事务与 ORM** | JPA 统一 | 需与 JPA 或混用 | 同 JPA 可搭配 |
| **复杂 SQL** | JPQL/原生 SQL 可用 | 更灵活 | 能力更强 |
| **运维与生态** | 普及 | 同 MySQL | 略小众但成熟 |

#### 选型评价

日志侧 **MySQL + JPA** 选型可行；与游戏服主存（MongoDB）的职责分工见 **2.6.1**。在无强需求（复杂报表、JSON 查询、分析库）前维持现状；若出现「重 SQL、轻 ORM」的模块，可局部引入 MyBatis；若需更强分析能力，可评估 PostgreSQL 作为读库或独立分析库。

### 2.8 缓存与 RPC 路由：Redis (Lettuce + Pool2)

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Redis + Lettuce (当前)** | 高 | 非阻塞、Spring Data 集成；连接池用 Commons Pool2，稳定。 |

#### 对比说明

- **Jedis**：传统阻塞 I/O，连接池成熟；在高并发下易成瓶颈，Spring Boot 2.x 起默认推荐 Lettuce，无特殊理由不回退。
- **Memcached**：仅 KV、无持久化与丰富数据结构；Redis 已覆盖缓存并承担 Stream 路由，无 Memcached 需求。

#### 横向对比小结

| 维度 | Redis + Lettuce (当前) | Jedis | Memcached |
|------|------------------------|-------|-----------|
| **I/O 模型** | 非阻塞 | 阻塞 | 阻塞 |
| **与 Spring/Netty** | 契合 | 可用但非首选 | 需单独集成 |
| **缓存 + Stream** | 同一产品，分工隔离 | 同左 | 无 Stream |
| **连接池** | Pool2，可调 | 内置池 | 通常内置 |
| **游戏服适用** | 高 | 中 | 仅纯 KV 时可选 |

#### 选型评价

**Redis + Lettuce** 作为业务缓存与 RPC 路由的选型合理：非阻塞、与现有栈一致，业务 Redis 与 RPC 路由 Redis 分离部署（如 docker/redis-route），职责清晰。RPC 转发中间件的进一步对比见 **2.8.1**。

### 2.8.1 RPC 转发中间件对比：Redis Stream vs Kafka / RocketMQ

跨服 RPC 需要将请求从调用方转发到目标服，并将响应回传。可选方案包括当前使用的 **Redis Stream**，以及 **Kafka**、**RocketMQ** 等消息中间件。下面从游戏服 RPC 转发的角度做横向对比。

#### 当前方案：Redis Stream（本项目）

- **模型**：按目标 serverId 使用独立 Stream key（`rpc:route:{serverId}`），请求 XADD 写入目标 Stream，目标服 XREADGROUP 消费；响应写入 `rpc:route:resp:{callerServerId}`，调用方同样用消费者组拉取。
- **特点**：无独立 Broker 进程，与业务 Redis 隔离（独立实例如 docker/redis-route）；实现简单、依赖少；Stream 支持 maxlen 裁剪（如 10000），防止目标离线时堆积；延迟低（P99 约 1ms 量级），适合实时 RPC。
- **局限**：吞吐受单 Redis 实例限制；无跨 Topic 的全局顺序、死信队列等企业级能力；持久化与回溯能力弱于专业 MQ。

#### Kafka

| 维度 | 说明 |
|------|------|
| **模型** | Topic + Partition；可把 serverId 作 key 或分区键，保证同一目标进同一分区，或一 Topic 多分区多消费者。 |
| **优势** | 高吞吐、持久化与多副本、消费位移管理、生态成熟（Connect、Streams）；适合海量消息与离线回溯。 |
| **劣势** | 部署与运维重（Broker + ZK/KRaft）；RPC 请求/响应通常需拆成两个 Topic 或用请求-响应关联 ID；冷启动与分区数规划对「按 serverId 路由」需额外设计。 |
| **适用** | 跨服 RPC 量极大、需持久化审计或与日志/数仓打通时，可考虑；一般游戏服 RPC 实时性优先，Kafka 偏重。 |

#### RocketMQ

| 维度 | 说明 |
|------|------|
| **模型** | Topic + Tag + 顺序消息；可一个 Topic 按 serverId 做 Tag 或 MessageQueue 路由。 |
| **优势** | 延迟与定时消息、重试/死信、事务消息；阿里系生态、中文文档与国内运维经验多。 |
| **劣势** | 同样需独立 Broker 集群；RPC 双向通信需自己维护请求-响应关联与超时。 |
| **适用** | 已有 RocketMQ 基础设施、或需要延迟/重试等能力时，可作为 Redis Stream 的替代或补充。 |

#### 横向对比小结

| 维度 | Redis Stream（当前） | Kafka | RocketMQ |
|------|----------------------|-------|----------|
| **部署与依赖** | 轻量，复用 Redis 或独立实例即可 | 需 Broker + 协调组件 | 需 Broker 集群 |
| **延迟** | 低（ms 级） | 通常 ms～十 ms | 通常 ms 级 |
| **吞吐** | 单实例约 1.5k～2k ops/s（与测试结果一致） | 单分区数万级 | 单队列万级 |
| **持久化与回溯** | 有 maxlen，非长期留存 | 强持久化、可回溯 | 可配置保留期 |
| **请求-响应模型** | 双 Stream（请求/响应）天然匹配 | 需自建关联与响应 Topic | 需自建关联与响应 Topic |
| **运维与学习成本** | 低 | 高 | 中高 |

#### 选型评价

- **当前以 Redis Stream 做 RPC 转发** 与项目阶段匹配：实现简单、无额外中间件、延迟与吞吐满足多 Game 服跨服调用；与直连 WebSocket 并存，按需启用 Redis 路由即可。
- **Kafka / RocketMQ** 更适合：消息量极大、需长期留存与审计、或与大数据/日志管道统一时再引入；届时可将「RPC 请求/响应」视为两种特殊 Topic，在现有 RpcRedisFacade / RedisRoutePublisher 同层抽象下替换为 Kafka/RocketMQ 生产者与消费者，而不必改动业务侧 RPC 接口。

### 2.9 协调与发现：Zookeeper + Curator

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Zookeeper + Curator (当前)** | 高 | 成熟、CP 一致性；当前用于服务注册与配置监听；Curator 5.x 与当前栈兼容。 |
| Nacos/Consul | 高 | 更偏配置与 HTTP 健康检查；若仅做服务发现可评估。 |
| etcd | 中 | K8s 生态；与现有 ZK 运维体系不同。 |

#### 对比说明

- **Zookeeper + Curator**：强一致（CP）；Curator 封装会话、重试、监听，与 Java 生态贴合；本项目当前用于「服务注册（临时节点做存活检测）」与「配置监听（CuratorCache 监听节点变更并增量更新 ZKConfig）」，ZK 模型（持久/临时节点、Watcher）匹配；如后续需要选主或分布式锁，Curator 的 LeaderSelector/LeaderLatch 也可直接扩展。
- **Nacos**：服务发现 + 配置中心一体，HTTP/OpenAPI 友好，适合与 Spring Cloud 或纯 HTTP 客户端集成；若希望「配置 + 发现」统一且偏 HTTP，可评估；与现有 ZK 的 CP 与会话模型不同。
- **Consul**：服务发现与健康检查、多数据中心；与 Nacos 类似，更偏运维与多机房；若未用 K8s，Consul 与 ZK 二选一即可。
- **etcd**：K8s 默认存储；若部署在 K8s 且希望用其做服务发现/配置，可考虑；与现有 ZK 运维体系并存会增加复杂度。

#### 横向对比小结

| 维度 | Zookeeper + Curator (当前) | Nacos | Consul | etcd |
|------|----------------------------|-------|--------|------|
| **一致性** | CP | AP/CP 可配 | CP | CP |
| **能力** | 协调、注册、配置监听（可扩展选主、锁） | 发现 + 配置 | 发现 + 健康检查 | 存储 + 选主 |
| **协议** | 自有二进制 | HTTP | HTTP/gRPC | gRPC |
| **与游戏服** | 注册、配置监听常用 | 配置中心友好 | 多机房友好 | K8s 内常用 |
| **运维** | 需 ZK 集群 | 相对轻量 | 需 Consul 集群 | 常随 K8s |

#### 选型评价

**Zookeeper + Curator** 满足当前服务注册与配置监听需求：CP 一致性可靠，临时节点做存活检测、CuratorCache 做配置变更监听已落地，Curator 5.x 与项目栈兼容。若后续需要选主或分布式锁，Curator 可直接扩展；若希望「配置中心」与「发现」统一、或全面上 K8s，可评估 **Nacos**（配置 + 发现）或 **etcd**（K8s 内）与 ZK 的职责划分，不必急于替换。

### 2.10 配置表：CSV + Commons CSV

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **CSV + Commons CSV (当前)** | 高 | 与 @Table、slg-support 加载逻辑一致；策划友好、版本可控。 |
| Excel 直接解析 | 中 | 需额外库与格式约定；CSV 更轻、易 diff。 |
| JSON/YAML 配置表 | 高 | 可读；与现有 CSV 流程与工具链不一致。 |

#### 对比说明

- **CSV + Commons CSV**：与 slg-support 的 @Table、表加载、CSV 路径约定一致；CSV 采用 Luban 格式（第 1 行为字段名，第 4 行起为数据行），由 `TableInjector`（BeanPostProcessor）扫描带 `@Table` 注解的字段，通过 `TableManager` 注册并加载对应 CSV，支持 `@TableId`（主键）、`@TableIndex`（索引）、`@TableRefCheck`（表关联校验）等注解；纯文本、Git 易 diff/merge；策划可用 Excel 编辑后另存为 CSV；无二进制格式，版本与评审清晰。
- **Excel 直接解析**：策划免「另存为 CSV」；需 POI 等依赖与格式规范（ sheet/列名/类型），大表解析与内存占用需注意；二进制不利于 diff。
- **JSON/YAML**：可读与嵌套好；与现有「按 CSV 行 → 表行」的加载逻辑不一致，需改 Table 加载与策划流程；适合少量复杂嵌套配置，不适合大批量表。

#### 横向对比小结

| 维度 | CSV + Commons CSV (当前) | Excel 直接解析 | JSON/YAML |
|------|-------------------------|----------------|-----------|
| **与 @Table 加载** | 已打通 | 需适配 | 需重写加载 |
| **策划工作流** | Excel 编辑→另存 CSV | 直接 Excel | 编辑器/导出 |
| **版本与 diff** | 文本，友好 | 二进制，差 | 文本，友好 |
| **大表与性能** | 流式/按行可做 | 需注意内存 | 视结构而定 |
| **嵌套结构** | 扁平为主 | 多 sheet 可表达 | 天然支持 |

#### 选型评价

**CSV + Commons CSV** 与现有表注解与加载流程匹配，策划与版本管理成本低，保持即可。若策划强依赖「只维护 Excel、不碰 CSV」，可评估 Excel 导出 CSV 的流水线或局部 Excel 解析，而不必全表改为 Excel 直读。

### 2.12 日志与检索：Logback + Elasticsearch（slg-log）

| 方案 | 可行性 | 对比要点 |
|------|--------|----------|
| **Logback + ES Java Client 8.x (当前)** | 高 | 集中日志与检索；官方客户端与 8.x 兼容。 |
| Logstash/Filebeat 采集 | 高 | 可与当前方案并存，做采集与缓冲。 |
| 仅文件日志 | 中 | 实现简单；无集中检索与分析能力。 |

#### 对比说明

- **Logback + Elasticsearch**：slg-log 通过 Appender 将日志写入 ES，便于集中检索与简单分析；ES 8.x 官方 Java Client 与当前版本对齐；需关注索引模板、生命周期与集群容量。
- **Loki**：与 Grafana 集成好、索引按标签、存储省；若已有 Grafana 体系可评估；与现有 ES 生态不同，迁移有成本。
- **仅文件 + 采集**：应用只写本地文件，由 Logstash/Filebeat 采集到 ES 或对象存储；解耦应用与存储、缓冲与重试在采集侧；当前若直写 ES 已稳定可维持，否则可引入 Filebeat 做缓冲。

#### 横向对比小结

| 维度 | Logback + ES (当前) | Loki + Grafana | 文件 + Filebeat → ES |
|------|---------------------|----------------|------------------------|
| **写入方式** | Appender 直写 ES | 推送/拉取到 Loki | 写文件，Beat 采集 |
| **检索** | Kibana/ES API | Grafana | 同 ES 或对象存储 |
| **依赖** | ES 集群 | Loki + Grafana | ES + Beat |
| **适用** | 已有 ES、需检索 | 已有 Grafana | 希望应用与存储解耦 |

#### 选型评价

**Logback + ES** 满足 slg-log 的集中日志与检索需求；需在运维侧落实索引生命周期与集群容量。若希望应用与存储解耦或减轻应用侧写入压力，可引入 **Filebeat** 做文件采集与缓冲，与现有 ES 并存。
