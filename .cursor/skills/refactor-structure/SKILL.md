# 结构重构（不改变逻辑）

## 输入

- **包**：包路径（如 `com.slg.game.develop.hero`），处理该包及**所有递归子包**下的类。
- **类**：单个类名或全限定类名，仅处理该类。

## 原则

- **不改变逻辑**：不修改业务行为、分支条件、返回值语义、对外接口；不新增或删除功能。
- **仅调整结构**：提取方法、拆分/合并类、调整包与职责划分、命名与格式，使结构更清晰、便于阅读与扩展。

## 目标

- **结构清晰**：职责单一、层次分明，相关逻辑内聚。
- **无过长方法/类**：方法过长时提取子方法；类职责过重或行数过多时拆分类或提取组件。
- **更好阅读体验**：命名表意、缩进与空行合理、复杂逻辑有注释或拆分为有语义的步骤。
- **易扩展**：便于在不破坏现有逻辑的前提下新增分支、策略或扩展点。

## 执行步骤

### 1. 确定范围与现状

- 根据输入解析目标包（递归子包）或唯一类，列出待重构的类。
- 快速扫描：识别过长方法（如单方法超 50～80 行或圈复杂度较高）、过长类（如超 300～500 行或职责混杂）、重复代码块、难以理解的命名或嵌套。

### 2. 方法级重构

- **过长方法**：按“做一件事”拆分为多个私有/包级方法，方法名体现步骤含义；原方法变为清晰的高层流程调用。
- **重复代码**：提取为私有方法或本类/工具方法，多处调用；不改变原有调用处的入参和结果。
- **复杂条件或计算**：提取为带语义的局部变量或私有方法（如 `isEligible()`、`computeScore()`），保持条件与结果一致。
- 保持方法签名与对外行为不变；仅新增内部方法时，不改变原有调用链的语义。

### 3. 类级重构

- **职责过重**：将可独立的一块职责提取为新的类（如策略、辅助、Builder），在原类中委托调用；对外接口与行为不变。
- **过长类**：按职责或“步骤阶段”拆成多个小类或组件，通过组合/委托与原逻辑等价；不改变对外的 API 与结果。
- **包结构**：若某包内类过多或职责混杂，可建议或实施子包划分（如按功能/层次），移动类时保证引用更新、逻辑不变。

### 4. 可读性与扩展性

- **命名**：变量、方法、类名表意清晰，与现有项目风格一致（参考 `.cursorrules` / `MEMORY`）。
- **注释**：对非自解释的复杂判断或业务规则补充简短中文注释，不写废话注释。
- **扩展点**：在保持逻辑不变的前提下，若明显存在“未来可能扩展”的点（如多种策略、不同实现），可适度引入接口或可替换组件，但不改变当前行为。

### 5. 校验与收尾

- **逻辑校验**：确认未修改分支条件、返回值、异常行为、对外接口；仅做了提取、重命名、拆分与移动。
- **构建**：确保 `mvn clean package` 通过；若有测试，保持通过。
- **收尾**：简要列出本次重构项（如“拆分了 X 方法”“提取了 Y 类”），并给出 1～2 条后续可扩展或可进一步优化的建议（可选）。

## 约束与注意

- 遵循项目 `.cursorrules` 与 `.cursor/MEMORY.md` 中的编码规范（类注释、日志、异常、命名等）。
- 不引入新的外部依赖（除非仅做已有依赖的合理使用方式调整）。
- 重构后单方法、单类规模建议：方法尽量控制在数十行内、类在数百行内，具体可结合项目既有风格微调。
- 若某处“逻辑与结构强耦合”、拆分会显著增加理解成本时，可保留局部略长但语义清晰的方法，并在注释中说明原因。

## 简要检查清单

- [ ] 输入已明确为包（递归子包）或唯一类
- [ ] 未改变任何业务逻辑与对外行为
- [ ] 过长方法已拆分或提取，过长/职责过重类已拆分或提取组件
- [ ] 命名与结构更清晰，可读性提升
- [ ] 构建通过，必要时测试通过
- [ ] 已做逻辑一致性检查并给出简要总结与改进建议
