# RPC 断线安全检查

## 目的

检查项目中所有 RPC 方法，分析在调用中途网络断开的场景下是否存在：
- 数据不一致（操作执行了但调用方不知道）
- 非幂等操作（重试导致重复执行）
- 异常未处理（CompletionException 未捕获导致线程异常）
- 状态残留（调用前修改了状态，RPC 失败后未回滚）

## 输入

- **无参数**：扫描项目中所有 `@RpcMethod` 接口及其调用点
- **指定接口**：仅检查指定的 RPC 接口（如 `IScenePlayerDataRpcService`）

## 检查模型

RPC 调用中途断线意味着：
1. **调用方**发出请求后连接断开，收到 `CompletionException`（超时或连接关闭）
2. **执行方**可能已完整执行方法并返回结果，但结果未送达调用方
3. 调用方无法区分"执行方没收到请求"和"执行方已执行但响应丢失"

因此每个 RPC 方法需满足：**即使执行方已成功执行，调用方重试时不会产生副作用（幂等性）。**

## 执行步骤

### 1. 扫描 RPC 接口

- 在 `slg-net/src/main/java/com/slg/net/rpc/impl/` 下查找所有带 `@RpcMethod` 注解的接口方法
- 记录每个方法的：接口名、方法名、参数、返回值、路由策略、执行模块

### 2. 查找实现类

- 在 `slg-scene`、`slg-game` 等业务模块中查找每个 RPC 接口的实现类（`@Component` 标注的实现）
- 记录实现类路径

### 3. 查找调用点

- 在所有模块中搜索每个 RPC 方法的调用点（通过 `@RpcRef` 注入的代理对象调用）
- 记录调用方法、所在类、调用上下文

### 4. 逐方法分析

对每个 RPC 方法，从以下维度分析：

#### 4.1 幂等性检查（执行方）

- 方法是否有副作用（修改数据库、修改内存状态、触发事件）？
- 如果调用方重试，副作用是否会重复？
- **安全**：查询类（无副作用）、先检查再操作（if not exists then create）、使用唯一键
- **危险**：无条件 insert（重复插入）、累加操作（重复加）、触发不可逆事件（发送奖励）

分析方法体中的操作：
```
对每个副作用操作，判断：
- 是否有前置检查防止重复执行？
- 是否使用了唯一约束防止重复写入？
- 重复执行的后果是什么？
```

#### 4.2 异常处理检查（调用方）

- 调用方是否捕获了 `CompletionException`？
- 使用 `.join()` 同步等待时，是否有 try-catch？
- 使用 `.whenComplete()` 异步回调时，是否处理了 error 参数？
- 异常发生后，调用方的状态是否一致？（如调用前已修改了本地状态，RPC 失败后是否回滚）

#### 4.3 状态一致性检查（调用方）

检查调用方在 RPC 前后的状态变更模式：

```
// 危险模式：先改状态再 RPC
player.setState(NEW_STATE);         // ← 已改
rpcService.doSomething(...).join(); // ← 失败了怎么办？

// 安全模式：RPC 成功后再改状态
int result = rpcService.doSomething(...).join();
if (result == SUCCESS) {
    player.setState(NEW_STATE);     // ← 确认成功后才改
}
```

#### 4.4 返回值语义检查

- `void` 返回（fire-and-forget）：调用方是否真的不关心执行结果？断线后是否需要补偿？
- `CompletableFuture<T>` 返回：调用方是否正确使用了返回值？

### 5. 生成报告

对每个 RPC 方法输出：

```
### 方法：IXxxRpcService.methodName
- 接口：IXxxRpcService（slg-net/.../xxx/）
- 实现：XxxRpcService（slg-scene/.../xxx/）
- 调用点：YyyService.method（slg-game/.../yyy/）

#### 幂等性：✅ 安全 / ⚠️ 需关注 / ❌ 不安全
- 分析：...

#### 异常处理：✅ 完备 / ⚠️ 部分缺失 / ❌ 未处理
- 分析：...

#### 状态一致性：✅ 安全 / ⚠️ 需关注 / ❌ 危险模式
- 分析：...

#### 修复建议（如有）：
- ...
```

### 6. 给出修复方案

对标记为 ⚠️ 或 ❌ 的方法，给出具体修复代码或设计建议：
- 幂等性修复：增加前置检查、使用唯一键、改为"查再改"模式
- 异常处理修复：补充 try-catch、补充回调 error 处理
- 状态一致性修复：调整状态变更时机（RPC 成功后再改）、增加回滚逻辑

## 约束

- 不修改任何代码逻辑，仅分析并输出报告和建议
- 遵循项目 `.cursorrules` 与 `.cursor/MEMORY.md` 中的规范
- 分析时考虑项目的线程模型（虚拟线程、多链/单链）
- `void` 返回的 RPC 是有意的 fire-and-forget 设计，不一定需要改为有返回值
