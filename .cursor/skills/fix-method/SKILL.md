# 根据注释补全方法

## 输入

- **包**：包路径（如 `com.slg.game.develop.hero`），处理该包及**所有递归子包**下的类。
- **类**：单个类名或全限定类名（如 `HeroService` 或 `com.slg.game.develop.hero.service.HeroService`），仅处理该类。

解析时若为简短类名，需在项目内唯一定位到具体类（必要时向用户确认）。

## 适用场景

| 场景 | 说明 | 动作 |
|------|------|------|
| **A** | 方法已有签名与 JavaDoc，方法体为空或桩（如 `throw new UnsupportedOperationException()`） | 按 JavaDoc 的说明、@param、@return、@throws 实现方法体 |
| **B** | 仅有类注释/模块注释，无方法或方法不完整 | 根据类注释推断职责，生成或补全方法声明 + JavaDoc + 实现 |
| **C** | 接口或抽象类已有方法声明与 JavaDoc，实现类未实现或未完全实现 | 在实现类中补全对应方法体，以接口/抽象类的 JavaDoc 为准 |

同一任务中可同时出现 A/B/C，按上述规则分别处理。

## 注释优先级与冲突处理

- **以 JavaDoc 为准**：实现必须符合方法或（在 C 中）接口/抽象类上 JavaDoc 的描述、参数、返回值、异常。若存在方法上方中文注释或行内 TODO，与 JavaDoc 冲突时以 JavaDoc 为准。
- **与现有实现冲突**：当已有方法体与 JavaDoc 不一致时，**以注释为准重写**该方法体，不保留与注释矛盾的逻辑。

## 实现约束

- **项目规范**：遵守仓库 `.cursorrules` 及 `.cursor/MEMORY.md` 中的编码规范（类注释、日志级别、异常处理、命名、包结构等）。
- **必要逻辑**：只实现注释所描述的逻辑，不添加注释未要求的额外功能、分支或对外行为。
- **复用优先**：优先调用**当前类已有方法**或**本类、父类、依赖中已存在的 API**；仅在必要时在本类内新增私有或包级辅助方法，且辅助方法也应有简短 JavaDoc（中文）。
- **性能**：避免在热点路径中不必要的对象创建、反射或重复计算；集合与 I/O 操作注意规模和资源释放。
- **可读性**：逻辑分层清晰，命名与注释一致；复杂判断可提取为带语义的局部变量或私有方法。

## 执行步骤

### 1. 确定范围

- 若输入为**包**：递归列出该包及子包下所有 `.java` 类（可根据需要排除仅作 DTO/常量/协议定义的类，或全部处理由用户指定）。
- 若输入为**类**：解析得到唯一目标类，仅处理该类。

### 2. 按场景识别

- 对每个目标类，识别需要补全的成员：
  - **A**：方法签名存在且方法体为空/桩/明显未完成。
  - **B**：类注释存在但缺少方法或方法明显不完整，需根据类职责推断。
  - **C**：类为某接口/抽象类的实现，存在未实现或默认/空实现的抽象方法，以父类型上的 JavaDoc 为准。

### 3. 补全实现

- 对每个待补全方法：
  - 以 **JavaDoc** 为唯一规范（场景 C 以接口/抽象类上的 JavaDoc 为准）。
  - 先检查本类及依赖中是否已有可复用方法，优先调用；再考虑本类内新增少量私有/包级辅助方法。
  - 编写实现，满足注释中的 @param、@return、@throws；不增加注释未描述的行为。
  - 遵循项目日志、异常、命名规范；注意性能和可读性。

### 4. 逻辑检查与建议

- **合理性**：检查空指针、边界条件、异常路径是否与 JavaDoc 一致；是否有明显逻辑错误或与注释矛盾。
- **收尾输出**：对本次修改做简短总结，并**主动给出 1～3 条**功能扩展或改进建议（如参数校验、可复用性、与现有模块的整合、性能或可读性优化等），不要求必须实现。

## 简要检查清单

- [ ] 输入已明确为包（递归子包）或唯一类
- [ ] 场景 A/B/C 已正确识别并分别按规则补全
- [ ] 所有实现以 JavaDoc 为准，冲突处已按注释重写
- [ ] 优先复用已有方法，仅必要时新增本类辅助方法
- [ ] 符合 .cursorrules / MEMORY 中的规范，无多余逻辑
- [ ] 已做逻辑合理性检查，并输出扩展/改进建议
